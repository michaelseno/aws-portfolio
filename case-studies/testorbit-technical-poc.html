TestOrbit — Technical POC

TestOrbit is a cloud-native test orchestration platform POC designed to run API, UI, performance, and security suites through a serverless control plane and containerized workers.

Problem

  • Teams need one workflow for manual, webhook, and scheduled test execution.
  • Execution should be isolated and scalable without managing permanent runner infrastructure.
  • Results and artifacts must be retrievable in a dashboard-friendly model.

Architecture

  • Control plane: API Gateway + Lambda handlers for suite, schedule, and run lifecycle.
  • Execution plane: ECS Fargate tasks launched per suite run.
  • Persistence: DynamoDB for suites, schedules, runs, and credit transactions.
  • Artifacts/reports: S3-backed report retrieval and result drill-down APIs.

Execution Flows

  • Manual run: dashboard action triggers suite run and worker task dispatch.
  • Webhook run: tokenized route dispatches execution for CI/CD integrations.
  • Scheduled run: EventBridge scheduler triggers periodic suite execution.
  • Result retrieval: summary and detail endpoints expose run status and outcomes.

Status and Next Steps

  • Status: Technical POC with production-oriented architecture patterns.
  • Implemented: core suite CRUD, run triggers, schedules, credits, reporting surfaces.
  • Next: transaction hardening, deeper observability, and deployment promotion workflow.

Proof and Scope Clarity

  • Proof: live frontend deployment and public frontend/backend repositories.
  • Proof: implemented run orchestration paths for manual, webhook, and schedules.
  • Current boundary: positioned as technical POC, not as a commercially launched SaaS.

What I'd Improve Next

  • Strengthen credit transaction consistency and scheduling state integrity.
  • Expand operational observability (alarms, traces, and failure runbooks).
  • Tighten API contract alignment and produce formal architecture docs for production handoff.
